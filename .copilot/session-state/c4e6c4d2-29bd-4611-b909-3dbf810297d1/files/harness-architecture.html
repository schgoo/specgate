Harness: test generation and execution

The harness reads a .spec.yaml file, generates language-specific test code from its cases, builds and runs the tests, and produces a structured JSON result. This document describes the pipeline, module decomposition, and key decisions.

01

Pipeline

The harness runs a seven-stage pipeline. Each stage has a single responsibility and a well-defined input/output contract.

1

Parse spec

Read and validate the .spec.yaml file against spec-schema.json. Produce a typed Spec struct with inputs, types, outcomes, outputs, and cases.

serde_yamlspec-schema.json
2

Resolve binding

Look up the binding file from binding: field (defaults to "mock" when absent). Validate against binding-schema.json. Extract language and project_root. Select backend from the language identifier in the backends registry.

binding-schema.json
2b

Prepare workdir

Create a unique scratch directory at rust/target/specgate-harness/{spec_name}/run-{timestamp}/. Generated test files and result JSON are written here. Unique suffixes prevent collisions between parallel runs.

isolation
3

Build & discover

Build the implementation project with the test-util feature flag enabled. Proc-macro annotations populate a compile-time registry of spec names → code symbols. The harness reads this registry to resolve which functions/methods implement each spec operation, setup, capture, etc.

compile-time registryproc macros
4

Generate test code

Select language backend from binding's language: field. For each spec case, emit a test function that constructs inputs, calls the implementation (using symbols from the registry), and asserts outputs. Write generated test file to the project (e.g. tests/specgate_generated.rs).

language-specific codegen
5

Build & run tests

Build and run the generated tests (e.g. cargo test --test specgate_generated). The backend knows the appropriate test command for each language.

processisolation
6

Collect JSON

Read each case's output JSON written by the generated tests. Aggregate per-case results with pass/fail status.

serde_json
7

Report

Produce structured JSON with per-case results, run metadata, and optional perf metrics. Write to stdout or a file.

JSON output
02

Data flow

How data moves through the pipeline, from spec file to JSON report.

The "Build & Discover" stage builds the implementation project and reads the compile-time annotation registry. Language backends plug into "Generate Tests". Everything else is shared.

03

Modules

The system is split across two crates in a Cargo workspace at rust/, plus language-specific backend crates.

specgate-types crate

Shared types used by all crates: SpecDocument, BindingFile, CaseResult, RunReport, RunError, RunOutcome. These are the serialized JSON contracts. No business logic.
spec_document.rs — SpecDocument, SpecCase binding_file.rs — BindingFile report.rs — CaseResult, RunReport, RunError, RunOutcome

specgate-harness crate

The harness library. Contains the Backend trait, the Harness orchestrator (pipeline entry point), and the MockBackend (behind test-util feature flag). No CLI yet — used as a library by tests.
backend.rs — Backend trait, Discovery, GeneratedArtifact harness.rs — Harness::run_spec() pipeline mock_backend.rs — MockBackend (test-util only)

specgate-rust-backend crate (planned)

Rust codegen backend. Implements the Backend trait to generate tests/specgate_generated.rs from specs and annotations.
rust_backend.rs — Backend impl for Rust

compare planned

Library for deep comparison of expected vs actual values. Used by generated tests for their assertions. Handles nested objects, lists (order-sensitive), oneof variant matching, extra fields (ignored), missing fields (mismatch).
compare.rs — deep_compare(expected, actual)

cli planned

clap-based CLI entry point. Not yet implemented — the harness is currently used as a library via Harness::new(repo_root).run_spec(path).
main.rs — CLI entry point
04

Backend trait

The codegen layer is pluggable. Each language backend implements this trait.

backend.rs (specgate-harness)
pub trait Backend: Send + Sync {
    /// Build the implementation project and discover annotations.
    /// Returns a Discovery with case metadata.
    fn build_and_discover(
        &self,
        binding: &BindingFile,
        spec: &SpecDocument,
    ) -> Result<Discovery, RunError>;

    /// Generate test file contents from a parsed spec and discovery.
    /// Writes files to the workdir and returns a GeneratedArtifact.
    fn generate(
        &self,
        binding: &BindingFile,
        spec: &SpecDocument,
        discovery: &Discovery,
        workdir: &Path,
    ) -> Result<GeneratedArtifact, RunError>;

    /// Execute the generated tests (e.g. cargo test).
    fn run_command(
        &self,
        binding: &BindingFile,
        generated: &GeneratedArtifact,
    ) -> Result<(), RunError>;

    /// Collect test results from JSON files written by generated tests.
    fn collect_results(
        &self,
        generated: &GeneratedArtifact,
    ) -> Result<Vec<CaseResult>, RunError>;
}

All methods take &BindingFile for project context and return unified RunError. Supporting types: Discovery (case metadata from build), GeneratedArtifact (paths to generated files + results). The Harness struct owns orchestration.

Key design: generated tests contain real assertions — they pass or fail with cargo test / dotnet test like any normal test. They also write structured JSON output so that separate tooling can collect results across languages for cross-language spec conformance comparison.

Capture wiring

Each annotation type maps to a specific codegen strategy for how the generated test captures its values.

Annotation
How the generated test captures it
spec_capture on struct
Read all public fields after operation returns, serialize to JSON object
spec_capture on fields
Read only annotated fields after operation returns, serialize to JSON object
spec_checkpoint (attribute)
Annotation macro records values to a thread-local log on each call; generated test drains the log after operation returns and includes in JSON output
spec_checkpoint! (inline)
Same thread-local log mechanism — macro evaluates the expression, records the value, returns it so execution continues normally
spec_mock
Generated test injects a mock that returns the value from the spec case's inputs, matched by mock name
spec_setup
Generated test calls the setup function with case input values to construct the object under test
06

Execution model

How test isolation, input delivery, and output collection work at runtime.

Concern
Approach
Case isolation
Each harness run gets a unique subdirectory under rust/target/specgate-harness/{spec_name}/run-{timestamp}/. Generated test files and result JSON are written there. No state leaks between runs.
Test generation
The backend generates test files into the workdir. For the mock backend, this is a single file. Real backends will generate into the project's test directory.
Build once, run all
Generated tests are built and run in one step (e.g. cargo test). The backend's run_command handles execution, collect_results gathers per-case JSON output.
Input delivery
Inputs are inlined as literals in the generated test code — the codegen backend constructs values directly from spec case data.
Output collection
Each generated test writes a JSON file to its case directory. After all tests run, the harness reads each case's JSON file and runs comparison.
07

Decisions

Decided and open architectural questions.

Generated tests assert and write JSON
Generated tests contain real assertions — they pass/fail natively with cargo test / dotnet test. They also write structured JSON output that separate tooling can use for cross-language conformance comparison (e.g. C# ↔ Spec ↔ Rust). The harness itself just generates and runs.
✓ Decided
One test file per spec, one function per case
All cases in a spec generate into a single test file. Each case is a separate test function named after the case. The file is placed in a well-known location so the harness knows where to find it.
✓ Decided
Build once, run all cases
The generated test file is compiled once. All test functions run in the same binary. This avoids per-case compilation overhead and is the natural model for both cargo test and dotnet test.
✓ Decided
Checkpoint capture via thread-local log
Both spec_checkpoint (attribute on method) and spec_checkpoint! (inline macro wrapping an expression) record values to a thread-local log. The generated test drains this log after the operation returns. This avoids modifying function signatures and works identically for both checkpoint forms.
✓ Decided
Per-run isolation via unique workdirs
Each harness run creates a unique subdirectory under rust/target/specgate-harness/{spec_name}/run-{timestamp}/. Generated test files and result JSON are written there. Unique suffixes prevent parallel test collisions. Workdirs are not cleaned automatically.
✓ Decided
Generated tests location (deferred)
The mock backend writes to the workdir scratch directory. The real Rust backend (planned) will write to tests/specgate_generated.rs in the target project so tests compile with the same toolchain and features. The exact location is a backend decision, not a core harness concern.
✓ Decided
Annotations provide the type bridge
The generated test does not need to construct arbitrary types from JSON. spec_setup functions accept spec input primitives and return the constructed object. spec_capture fields provide output values. spec_checkpoint records intermediates via thread-local log. No Deserialize or type-mapping table is required — the annotations ARE the bridge between spec values and implementation types.
✓ Decided
Compile-time registry for name → symbol resolution
The binding file does not declare which functions to call. Instead, proc-macro annotations register metadata (spec name, kind, code path) into a compile-time registry. The harness builds the implementation project first, reads the registry to discover name → symbol mappings, then generates test code that calls the resolved symbols directly. This keeps the binding minimal (language + project_root) and makes annotations the single source of truth for both instrumentation and code discovery.
✓ Decided
Perf metrics and code coverage collection
The spec declares PerfMetrics and CoverageReport as optional outputs, but the mechanism for collecting them is unspecified. Perf metrics likely need Gungraun/Valgrind wrapping. Coverage needs llvm-cov or dotnet-coverage. These could be separate bindings, CLI flags, or post-processing steps.
○ Open

CLI interface

The harness is invoked as a CLI tool.

Usage
specgate run \
  --spec specs/parser.spec.yaml \
  --binding rust \
  --workdir /tmp/specgate-run \
  --output results.json

# Pass-through mode (skip codegen, run command directly)
specgate run \
  --spec specs/parser.spec.yaml \
  --mode passthrough \
  --workdir /tmp/specgate-run
Flag
Description
--spec <path>
Path to the .spec.yaml file (required)
--binding <name>
Binding identifier — resolves to bindings/<name>.yaml. Overrides the spec's binding: field.
--bindings-dir <dir>
Directory containing binding files (default: bindings/)
--workdir <dir>
Working directory for execution and {workdir} resolution (default: temp dir)
--output <file>
Write JSON results to file instead of stdout
--case <name>
Run only this case (for debugging a single failure)
--verbose
Print build and run output to stderr
